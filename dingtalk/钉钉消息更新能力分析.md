# 钉钉消息更新能力分析

## 当前实现 vs 理论能力

| 功能 | 飞书（已实现） | 钉钉当前（MVP） | 钉钉理论能力 | 实现难度 |
|------|---------------|----------------|-------------|---------|
| **实时进度更新** | ✅ 完整支持 | ❌ 不支持 | 🟢 可以实现 | ⭐⭐⭐ |
| **卡片样式** | ✅ 彩色卡片头 | 🟡 Emoji 前缀 | 🟢 互动卡片 | ⭐⭐ |
| **消息更新** | ✅ updateCard | ❌ 每次新消息 | 🟢 updateInteractiveCard | ⭐⭐⭐ |
| **Markdown 支持** | ✅ 完整 | ✅ 完整 | ✅ 完整 | - |
| **语音识别** | ✅ 火山引擎 | 🟡 本地 whisper | 🟢 可接入火山 | ⭐ |

## 详细分析

### 1. 实时进度更新（❌ → 🟢）

**当前状态：**
```typescript
// 钉钉：每次进度都是新消息
await sendMarkdown(webhook, '⏳ Cursor AI 正在思考...');
// ... agent 执行 ...
await sendMarkdown(webhook, '✅ 执行完成：结果内容');
```

**理论实现：**
```typescript
// 1. 首次发送互动卡片，获取 cardInstanceId
const cardInstanceId = await sendInteractiveCard(webhook, {
  outTrackId: 'task-' + Date.now(),
  cardTemplateId: 'xxx',
  cardData: { status: '思考中...', progress: 0 }
});

// 2. 实时更新卡片内容
await updateInteractiveCard(cardInstanceId, {
  cardData: { status: '执行工具...', progress: 50 }
});

// 3. 完成时更新
await updateInteractiveCard(cardInstanceId, {
  cardData: { status: '完成', progress: 100, result: '...' }
});
```

**关键 API：**
- **发送卡片**：`POST /v1.0/card/instances/createAndDeliver`
- **更新卡片**：`PUT /v1.0/card/instances`
- **参数**：
  - `outTrackId`：卡片唯一标识（开发者生成）
  - `cardTemplateId`：卡片模板 ID（需在开放平台创建）
  - `cardData`：卡片数据（可动态更新）

### 2. 互动卡片 vs Markdown 消息

#### 当前 Markdown 方案：
**优点：**
- ✅ 实现简单，无需额外配置
- ✅ 支持丰富的 Markdown 语法
- ✅ 无需创建卡片模板

**缺点：**
- ❌ **无法更新**，只能发送新消息
- ❌ 样式简单，只有纯文本 + Emoji
- ❌ 历史消息多，聊天记录混乱
- ❌ 无法显示进度条、按钮等交互组件

#### 互动卡片方案：
**优点：**
- ✅ **支持实时更新**（同一条消息）
- ✅ 丰富的样式（进度条、按钮、表单）
- ✅ 清爽的聊天记录（只有一条消息）
- ✅ 支持用户交互（点击按钮触发回调）

**缺点：**
- ⚠️ 需要在钉钉开放平台创建卡片模板
- ⚠️ 模板审核需要时间
- ⚠️ 代码复杂度提升（模板管理、卡片数据映射）
- ⚠️ 调试困难（需要在钉钉客户端查看效果）

### 3. 技术限制对比

| 限制项 | 飞书 | 钉钉 |
|-------|------|------|
| **卡片大小** | 30KB (~3500 汉字) | 32KB |
| **Markdown 表格** | 最多 5 个 | 无限制 |
| **消息更新频率** | 无明确限制 | 建议 < 10次/秒 |
| **卡片模板** | 动态生成 | 需预先创建 |
| **模板审核** | 不需要 | 需要审核 |

### 4. 实现难度评估

#### 方案 A：升级到互动卡片（⭐⭐⭐ 中等）

**工作量：**
1. ⏱️ **2-3 小时**：在钉钉开放平台创建卡片模板
   - 进度卡片模板（显示思考/执行/完成状态）
   - 结果卡片模板（显示执行结果）
   - 等待模板审核通过（1-3 个工作日）

2. ⏱️ **4-6 小时**：重构消息发送逻辑
   ```typescript
   // 新增函数
   async function sendInteractiveCard(...)
   async function updateInteractiveCard(...)
   
   // 修改 handleMessage 流程
   const cardId = await sendInteractiveCard(...); // 首次发送
   
   // runAgent 回调中更新进度
   onProgress: (p) => {
     updateInteractiveCard(cardId, { progress: p.progress });
   }
   ```

3. ⏱️ **2-3 小时**：测试和调试
   - 测试卡片发送和更新
   - 处理错误情况（卡片更新失败等）
   - 调整卡片样式

**总计：8-12 小时**

#### 方案 B：保持 Markdown 并优化（⭐ 简单）

**工作量：**
- ⏱️ **1 小时**：增强 Markdown 样式
  - 更丰富的 Emoji 前缀
  - 更好的进度信息格式化
  - 更清晰的分隔符

**总计：1 小时**

### 5. 推荐方案

#### 短期（当前 MVP 阶段）：
**保持 Markdown 方案** ⭐

**理由：**
- ✅ 已经能满足基本需求
- ✅ 开发和维护成本低
- ✅ 无需依赖平台审核
- ✅ 与飞书功能对齐（都支持基本消息）

**优化建议：**
```typescript
// 可以考虑添加进度日志折叠功能
await sendMarkdown(webhook, `
⏳ **任务执行中**

**进度：** 50%
**当前阶段：** 🔧 执行工具
**已用时间：** 12s

---
<details>
<summary>查看详细日志</summary>

- [12:00:01] 开始思考
- [12:00:05] 调用 Read 工具
- [12:00:10] 调用 StrReplace 工具
</details>
`);
```

#### 中期（v3.1+）：
**升级到互动卡片** ⭐⭐⭐

**时机：**
- 用户反馈强烈需要实时更新
- 有时间投入开发和调试
- 卡片模板审核通过

**收益：**
- ✅ 极大提升用户体验
- ✅ 减少消息刷屏
- ✅ 更专业的展示效果
- ✅ 与飞书体验对齐

## 结论

**当前能力：** 🟡 基本满足，但无实时更新

**理论能力：** 🟢 完全可以实现飞书级别的体验

**建议：** 
1. **现阶段**继续使用 Markdown（快速迭代，稳定可靠）
2. **积累用户反馈**，评估实时更新的真实需求
3. **v3.1 版本**考虑升级到互动卡片（提升用户体验）
