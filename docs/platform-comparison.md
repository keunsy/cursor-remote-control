# 飞书 vs 钉钉能力对比

## 核心差异总结

| 能力 | 飞书 | 钉钉 | 影响 |
|------|------|------|------|
| **消息更新** | ✅ 支持 `updateCard` | ❌ 不支持 | 🔴 **重大** |
| **主动推送** | ✅ 可发送到 chatId | ⚠️ 仅通过 webhook | 🟡 **中等** |
| **交互卡片** | ✅ 丰富交互 | ⚠️ ActionCard 有限 | 🟡 **中等** |
| **流式传输** | ✅ WebSocket | ✅ Stream | 🟢 **无影响** |
| **文件处理** | ✅ 完整支持 | ✅ 完整支持 | 🟢 **无影响** |

---

## 详细对比

### 1. 消息更新能力（最关键差异）

#### 飞书
```typescript
// 可以多次更新同一张卡片
await updateCard(cardId, "🤔 思考中...");
await updateCard(cardId, "🔧 执行工具...");  // 更新同一张卡片
await updateCard(cardId, "✅ 完成");          // 再次更新
```

**实现的功能：**
- ✅ 实时进度显示（思考中 → 执行工具 → 回复中）
- ✅ 排队状态提示（"当前有任务在进行中，正在排队..."）
- ✅ 重新组织内容（超限时提示"正在重新组织..."）
- ✅ 单卡片体验（用户只看到一张卡片，不刷屏）

#### 钉钉
```typescript
// 只能发送新消息
await sendMarkdown(webhook, "🤔 思考中...");      // 第1条消息
await sendMarkdown(webhook, "🔧 执行工具...");     // 第2条消息（刷屏）
await sendMarkdown(webhook, "✅ 完成");           // 第3条消息（刷屏）
```

**无法实现：**
- ❌ 实时进度更新（会刷屏）
- ❌ 排队状态提示（发了消息就撤不回）
- ❌ 优雅的错误处理（超限重试等）

**解决方案：**
- 只发送最终结果（不显示中间进度）
- 或者接受刷屏（用户体验较差）

---

### 2. 主动推送能力

#### 飞书
```typescript
// 可以主动发送到任意 chatId
await sendCard(chatId, "定时提醒：该喝水了");
```

- ✅ 定时任务随时推送
- ✅ 心跳检查主动提醒
- ✅ 跨会话通知

#### 钉钉
```typescript
// 只能通过 webhook 回复，webhook 是动态的
const webhook = getWebhook();  // 需要用户先发过消息
if (!webhook) {
    console.warn("无活跃 webhook，无法推送");
    return;
}
await sendMarkdown(webhook, "定时提醒：该喝水了");
```

**限制：**
- ⚠️ 用户必须先发送至少一条消息（建立 webhook 缓存）
- ⚠️ Webhook 有时效性（可能失效）
- ⚠️ 无法跨会话推送

**解决方案：**
- 缓存最近活跃的 webhook
- 推送失败时记录日志（静默失败）
- 心跳功能受限（需要用户先互动）

---

### 3. 交互卡片能力

#### 飞书
```typescript
// 支持按钮、表单、多列布局等
{
    "tag": "div",
    "elements": [
        {
            "tag": "button",
            "text": "切换项目",
            "type": "primary",
            "value": "switch_project"
        }
    ]
}
```

- ✅ 按钮交互
- ✅ 表单填写
- ✅ 复杂布局

#### 钉钉
```typescript
// ActionCard 功能有限
{
    "msgtype": "actionCard",
    "actionCard": {
        "title": "标题",
        "text": "markdown 内容",
        "singleTitle": "按钮",
        "singleURL": "链接"
    }
}
```

- ⚠️ 仅支持单个按钮 + 链接
- ⚠️ 无法实现复杂交互
- ⚠️ Markdown 渲染能力有限

**影响：**
- 会话管理、项目切换等需要简化交互
- 无法实现复杂的 UI 界面

## 推荐策略

### 飞书适合
- 需要**实时反馈**的场景（看到 AI 思考过程）
- 长时间任务（需要排队提示、进度更新）
- 复杂交互（按钮、表单等）
- 主动推送场景（心跳、定时提醒）

### 钉钉适合
- 只关心**最终结果**的场景
- 简单问答（不需要看中间过程）
- 文件处理、代码生成等（过程不重要）
- 已有互动的用户（定时任务可用）

---

## 技术限制的根本原因

### 飞书
- 企业级 IM，API 设计完善
- 支持卡片 ID 引用和更新
- 主动推送能力强

### 钉钉
- 基于 webhook 的回复机制
- Webhook 是一次性的、动态的
- 无法引用和更新已发送的消息
- 安全考虑：防止机器人滥用

---

## 最终建议

**不要强求钉钉实现飞书的所有功能。**

钉钉的定位是：
- ✅ 基础 AI 对话（问答、代码、文件处理）
- ✅ 定时任务（需用户先发消息）
- ❌ 实时进度显示
- ❌ 复杂交互界面
- ⚠️ 心跳检查（受限）

如果用户需要**实时反馈和复杂交互**，推荐使用飞书。
钉钉更适合**结果导向、轻量级**的使用场景。
