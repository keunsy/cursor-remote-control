### 钉钉版（待启动 ⏳）

#### 2.1 获取钉钉凭据（20分钟）

参考：`~/work/cursor/dingtalk-cursor-claw/钉钉应用配置指南.md`

**快速步骤**：
1. 访问 https://open-dev.dingtalk.com/
2. 创建企业内部应用
3. 添加机器人能力 → **选择 Stream 模式** ⚠️
4. 配置权限：机器人消息接收、内容读取、文件下载
5. 事件订阅：`/v1.0/im/bot/messages/get`（Stream 模式）

```bash
cd ~/work/cursor/dingtalk-cursor-claw
vim .env
```

修改这两行：

```bash
DINGTALK_APP_KEY=dingxxxxxx          # 填你的 AppKey
DINGTALK_APP_SECRET=xxxxxxxxxxxxx    # 填你的 AppSecret
```

其余保持不变（CURSOR_MODEL=auto 等）。

#### 2.3 启动服务（1分钟）

**方式 A：前台测试**（推荐先用这个）

```bash
bun run server-minimal.ts

看到以下输出说明成功：

│  钉钉 → Cursor Agent 中继服务 v1 (MVP)           │
│  ...
└──────────────────────────────────────────────────┘

[钉钉] access_token 已刷新
钉钉 Stream 连接已建立，等待消息...
```

**方式 B：后台服务**（验证成功后）

```bash
bash service.sh install
```

开机自启动 + 崩溃自动恢复。

#### 2.4 测试对话（30秒）

在钉钉中 @机器人：

## 六、符号链接说明

### 什么是符号链接

符号链接 = Windows 快捷方式 = macOS 替身

```bash
飞书版/bridge.ts    ← 真实文件（10KB）
                    ↑
钉钉版/bridge.ts ───┘  符号链接（0KB，只是指针）
```

### 实际效果

```bash
# 查看
$ ls -la ~/work/cursor/dingtalk-cursor-claw/bridge.ts
lrwxr-xr-x  bridge.ts -> ../feishu-cursor-claw/bridge.ts
                         └─ 箭头表示"指向"

# 读取（自动跳转）
$ cat ~/work/cursor/dingtalk-cursor-claw/bridge.ts
# 实际读取：~/work/cursor/feishu-cursor-claw/bridge.ts

# 修改飞书版
$ vim ~/work/cursor/feishu-cursor-claw/bridge.ts
# 钉钉版自动同步（因为它们是同一个文件）
```

### 优势

✅ **代码零重复**：核心逻辑只存一份  
✅ **自动同步**：改一处两边生效  
✅ **磁盘节省**：符号链接几乎不占空间  
✅ **维护简单**：Bug 修复一次搞定

### 共享的核心模块

- `bridge.ts` - Cursor CLI 调用（复用）
- `memory.ts` - 记忆向量搜索（复用）
- `heartbeat.ts` - 心跳检查（复用）
- `scheduler.ts` - 定时任务（复用）
- `memory-tool.ts` - 记忆工具（复用）
- `backfill-embeddings.ts` - 索引工具（复用）
- `sync-apple-notes.ts` - 备忘录同步（复用）

**复用率：70%**

### 常见错误

| 错误 | 原因 | 解决 |
|------|------|------|
| `clientId or clientSecret is null` | .env 未配置钉钉凭据 | 填入 DINGTALK_APP_KEY/SECRET |
| `access_token 获取失败` | AppKey/Secret 错误 | 检查钉钉后台的凭据 |
| `收不到消息` | 事件订阅未配置 | 检查是否订阅了机器人消息 |
| `Stream 连接失败` | 网络问题或应用未发布 | 检查网络和应用状态 |

## 八、核心问题解答

### Q1: 代码是拷贝的吗？

**A: 不是！** 核心代码用**符号链接**共享。

```
bridge.ts, memory.ts, heartbeat.ts, scheduler.ts 
→ 符号链接指向飞书版（同一份代码）

server-minimal.ts, dingtalk-client.ts
→ 真实文件（钉钉专属）
```

**磁盘占用对比**：
- 完全复制：130MB（两份完整代码）
- 符号链接：70MB（核心代码只存一份）

### Q2: 启动也要两次吗？

**A: 是的，但这是必要的。**

```
进程 1 (飞书): 连接飞书 WebSocket 服务器
进程 2 (钉钉): 连接钉钉 Stream 服务器
         ↓
    同一个 Cursor CLI
         ↓
    共享记忆和配置
```

**为什么必须两个进程？**
- 飞书和钉钉是完全不同的协议
- 无法用一个连接同时接入两个平台
- 隔离性：一个崩溃不影响另一个

**资源占用**：
- 内存：~300MB（两个进程合计）
- 对比：Chrome 开 5 个 Tab 就超过这个了
- 可以忽略不计

### Q3: 两个服务会冲突吗？

**A: 不会！** 它们完全隔离：

| 维度 | 飞书服务 | 钉钉服务 |
|------|---------|---------|
| 进程 | PID 76806 | PID 独立 |
| 服务名 | com.feishu-cursor-claw | com.dingtalk-cursor-claw |
| 日志 | /tmp/feishu-cursor.log | /tmp/dingtalk-cursor.log |
| 会话key | feishu_xxx | dingtalk_xxx |
| IM连接 | 飞书 WebSocket | 钉钉 Stream |

**共享的资源（并发安全）**：
- `projects.json`（只读配置，无冲突）
- `.cursor/MEMORY.md`（SQLite 并发安全）
- 核心代码（符号链接，只读）

### Q4: 修改代码会影响另一个吗？

**A: 看情况。**

**修改核心模块（bridge.ts、memory.ts等）**：
- 飞书和钉钉**都会**更新（符号链接共享）
- 适合：Bug 修复、功能增强

**修改平台专属文件**：
- 只影响当前平台
- 适合：钉钉特色功能、飞书特色功能

**示例**：

```bash
# 修改共享模块（两边都更新）
vim ~/work/cursor/feishu-cursor-claw/bridge.ts
→ 钉钉版自动更新（符号链接）

# 修改钉钉专属（只影响钉钉）
vim ~/work/cursor/dingtalk-cursor-claw/server-minimal.ts
→ 飞书版不受影响
```
