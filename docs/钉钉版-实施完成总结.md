## 回答你的担心

### 1. "代码岂不是都要拷贝一份？"

❌ **不是完全拷贝！** 通过**符号链接**共享核心代码：

```
飞书版/bridge.ts     ← 真实文件（10KB）
钉钉版/bridge.ts     ← 符号链接（几乎 0KB，只是指针）
                      ↑ 指向飞书版
```

**效果**：
- 修改飞书版 bridge.ts，钉钉版自动同步
- 核心代码只存一份
- Bug 修复一处生效

**磁盘对比**：
- 完全复制：100MB
- 符号链接：70MB（node_modules 是独立的，核心代码共享）

### 2. "启动的话也得多启动？"

✅ **是的，需要两个进程，但这是必要的：**

```
进程 1 (PID 76806): feishu-cursor-claw   ← 飞书 WebSocket
进程 2 (PID xxxxx): dingtalk-cursor-claw ← 钉钉 Stream
            ↓
      同一个 Cursor CLI
```

**为什么必须两个？**
- 飞书和钉钉是完全不同的协议（WebSocket vs Stream）
- 无法用一个进程同时连接两个平台
- 隔离性好：一个挂了不影响另一个
- 并发性好：可以同时处理两边消息

**资源占用**：
- 每个进程 ~150MB 内存
- 总共 ~300MB
- Chrome 几个 Tab 就超过这个了

**自动管理**：
```bash
# 一次配置，永久生效
bash ~/work/cursor/manage-both-services.sh install

# 开机自启动，崩溃自动恢复，无需手动管理
```
