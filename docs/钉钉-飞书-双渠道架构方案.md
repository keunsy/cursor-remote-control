## 实现策略

### 策略 1：符号链接共享代码（推荐）✅

```bash
cd ~/work/cursor/dingtalk-cursor-claw

# 创建符号链接，指向飞书版的核心模块
ln -s ../feishu-cursor-claw/bridge.ts bridge.ts
ln -s ../feishu-cursor-claw/memory.ts memory.ts
ln -s ../feishu-cursor-claw/memory-tool.ts memory-tool.ts
ln -s ../feishu-cursor-claw/heartbeat.ts heartbeat.ts

# scheduler 可能需要独立（避免两个服务同时调度同一任务）
cp ../feishu-cursor-claw/scheduler.ts scheduler.ts
```

**优点**：
- 代码 100% 同步
- 修复 bug 一处改两边生效
- 磁盘占用小

**缺点**：
- 删除飞书版会影响钉钉版

### 策略 2：直接复制（备选）

```bash
cp -r feishu-cursor-claw dingtalk-cursor-claw
cd dingtalk-cursor-claw
# 然后改 server.ts 和 .env
```

**优点**：
- 完全独立，互不影响
- 可以各自演化

**缺点**：
- 代码重复
- Bug 修复需要两边同步

### 策略 3：抽取共享库（长期方案）🎯

```bash
~/work/cursor/
├── cursor-bridge-core/              # 核心共享库（新建）
│   ├── bridge.ts

│   ├── scheduler.ts

├── feishu-cursor-claw/              # 飞书服务
│   ├── server.ts                    # 只负责飞书协议
│   ├── package.json                 # 依赖: cursor-bridge-core
│   └── ...
│
└── dingtalk-cursor-claw/            # 钉钉服务
    ├── server.ts                    # 只负责钉钉协议
    ├── package.json                 # 依赖: cursor-bridge-core
```

**优点**：
- 最优雅

**缺点**：
- 需要重构现有代码

**定时任务处理**：两种方式任选

**方式A：共享定时任务（推荐）**
- 两个服务读取同一个 `cron-jobs.json`
- 用分布式锁防止重复执行（基于文件锁或 SQLite）
- 简单场景下可以接受轻微的竞态（同一任务被触发两次问题不大）

**方式B：独立定时任务**
- 飞书服务：`feishu-cron-jobs.json`
- 钉钉服务：`dingtalk-cron-jobs.json`
- AI 创建任务时可以选择渠道

### 🎯 Phase 2: 长期优化（可选）

稳定运行一段时间后，如果觉得维护两份代码麻烦，再重构为**策略3（共享库）**。

## 会话隔离策略

两个服务的会话 key 使用不同前缀：

```typescript
// feishu-cursor-claw/server.ts
const sessionKey = `feishu_${conversationId}_${userId}`;
```

### 3. 定时任务（两种方案）

**方案A：共享 cron-jobs.json + 分布式锁**

```typescript
// scheduler.ts 改造
import { open } from 'better-sqlite3';

class Scheduler {
  private tryAcquireLock(jobId: string): boolean {
    // 使用 SQLite 作为分布式锁
    const db = open('.scheduler-lock.db');
    try {
      db.exec(`
        CREATE TABLE IF NOT EXISTS locks (
          job_id TEXT PRIMARY KEY,
          service TEXT,
          locked_at INTEGER
        )
      `);
      
      const now = Date.now();
      db.exec(`
        INSERT INTO locks (job_id, service, locked_at) 
        VALUES ('${jobId}', '${this.serviceName}', ${now})
      `);
      return true;
    } catch {
      return false; // 已被锁定
    }
  }
  
  async executeJob(job: Job) {
    if (!this.tryAcquireLock(job.id)) {
      console.log(`Job ${job.id} already locked by another service`);
      return;
    }
    
    // 执行任务...
    
    // 释放锁
    this.releaseLock(job.id);
  }
}
```

**方案B：独立任务文件**

```bash
# 飞书版配置
feishu-cursor-claw/.env
CRON_JOBS_FILE=../cron-jobs-feishu.json

# 钉钉版配置

CRON_JOBS_FILE=../cron-jobs-dingtalk.json
```

AI 创建任务时可以智能选择：

- 从钉钉发起 → 写入 `cron-jobs-dingtalk.json`
- 或者用户明确指定渠道

```typescript
  clientId: process.env.DINGTALK_APP_KEY,
  clientSecret: process.env.DINGTALK_APP_SECRET,
  subscriptions: [
    { type: 'CALLBACK', topic: '/v1.0/im/bot/messages/get' }

client.registerCallbackListener(
  '/v1.0/im/bot/messages/get',
  async (event) => {

);
```

### 定时任务（推荐分开）

**推荐：独立任务文件**

```bash
~/work/cursor/
├── cron-jobs-feishu.json     # 飞书创建的任务
└── cron-jobs-dingtalk.json   # 钉钉创建的任务
```

修改 scheduler.ts：

```typescript
// 根据环境变量选择任务文件
const CRON_FILE = process.env.CRON_JOBS_FILE || 
  `../cron-jobs-${process.env.SERVICE_NAME || 'feishu'}.json`;
```

**为什么推荐分开**：
- 避免加锁的复杂度
- 清晰知道任务来自哪个渠道
- 任务执行结果发送到正确的平台

**如果一定要共享**：
- 在任务对象中加 `source: 'feishu' | 'dingtalk'` 字段
- 执行时两个服务通过文件锁协调（见上面的实现）

## 注意事项

### 1. 会话隔离

```typescript
// 确保 sessionKey 带平台前缀
const sessionKey = `dingtalk_${conversationId}_${senderId}`;
```

### 2. 心跳分开

```typescript
// heartbeat-state.json 分平台存储
feishu-heartbeat-state.json
dingtalk-heartbeat-state.json
```

或者在同一文件中分 key：

```json
{
  "feishu": {
    "lastChecks": { ... },
    "lastHeartbeat": "2026-03-11T16:00:00Z"
  },
  "dingtalk": {
    "lastChecks": { ... },
    "lastHeartbeat": "2026-03-11T16:05:00Z"
  }
}
```

### 3. 错误隔离

一个服务挂了不影响另一个：

```typescript
// 在各自的 server.ts 中
process.on('uncaughtException', (error) => {
  console.error('[DingTalk Service] Uncaught exception:', error);
  // 记录日志但不退出
});
```

### 4. 资源竞争

**可能的竞争点**：
- ✅ 记忆文件：SQLite 自带锁，追加写入安全
- ✅ 项目配置：只读，无竞争
- ⚠️ 定时任务：建议分开或加锁
